## Logic

### ADR-005 (committed snapshot vs live fetch)

Every mobile build needs TypeScript types for the backend's API surface. Two patterns:

| Pattern | Build depends on backend running? | Reproducibility | Drift risk |
|---------|:---:|:---:|:---:|
| Live fetch at build time | Yes | Depends on backend version | High — builds succeed against dev backend but break on CI / production |
| Snapshot committed + explicit refresh | No | Deterministic | Low — types lag real API only until explicit refresh |

Phase 187 picks **snapshot**. Rationale, triggers, and revisit conditions in `docs/adr/005-openapi-spec-snapshot.md`. Key practical consequences:

- `api-schema/openapi.json` is the committed source of truth for typed client generation.
- `npm run refresh-api-schema` is the explicit trigger when Kerwyn updates the backend and wants mobile types to catch up.
- Mobile build is self-contained — works on planes, at coffee shops, without the backend running.
- Developer workflow: edit backend → `cd moto-diag && .venv/Scripts/python.exe -m motodiag serve` → in mobile repo `npm run refresh-api-schema && npm run generate-api-types` → types updated → mobile code refactors happen on the new types.

### Type generation pipeline

```
moto-diag backend /openapi.json  (Phase 183 enriched)
            │
            ▼   npm run refresh-api-schema
      api-schema/openapi.json  (committed)
            │
            ▼   npm run generate-api-types  (openapi-typescript)
      src/api-types.ts  (committed)
            │
            ▼   import type { paths } from './api-types'
      src/api/client.ts  (openapi-fetch createClient<paths>)
```

`src/api-types.ts` is a single generated file with the `paths` + `components` exports. Pinned version: `openapi-typescript@^7.x`. Output is stable across runs given identical input (good for reviewable diffs).

### Client shape

```ts
// src/api/client.ts (real, Phase 187)
import createClient from 'openapi-fetch';
import type { paths } from '../api-types';
import Config from 'react-native-config';
import { getApiKey } from './auth';

export interface ApiClientOptions {
  baseUrl?: string;
}

export function makeClient(options: ApiClientOptions = {}) {
  const baseUrl =
    options.baseUrl ?? Config.API_BASE_URL ?? 'http://10.0.2.2:8000';

  const customFetch: typeof fetch = async (input, init) => {
    const apiKey = await getApiKey();
    const headers = new Headers(init?.headers);
    if (apiKey) headers.set('X-API-Key', apiKey);
    headers.set('Accept', 'application/json');
    return fetch(input, { ...init, headers });
  };

  return createClient<paths>({ baseUrl, fetch: customFetch });
}

// Module-level singleton for app-wide use.
export const api = makeClient();
```

Caller use:

```ts
const { data, error, response } = await api.GET('/v1/vehicles');
//      ^? VehicleListResponse | undefined
//                     ^? ProblemDetail | undefined
if (error) return handleProblemDetail(error);
console.log(data.items.length);
```

### Auth storage

`react-native-keychain` provides:
- Android: Android Keystore-backed secure storage with hardware-TEE encryption on most devices from Android 6.0+.
- iOS: Keychain Services (Secure Enclave on supported devices).

Wrapper in `src/api/auth.ts`:

```ts
import * as Keychain from 'react-native-keychain';

const KEYCHAIN_SERVICE = 'moto-diag-mobile';
const KEYCHAIN_USERNAME = 'api-key';

export async function getApiKey(): Promise<string | null> {
  const result = await Keychain.getGenericPassword({ service: KEYCHAIN_SERVICE });
  return result ? result.password : null;
}

export async function setApiKey(key: string): Promise<void> {
  await Keychain.setGenericPassword(KEYCHAIN_USERNAME, key, { service: KEYCHAIN_SERVICE });
}

export async function clearApiKey(): Promise<void> {
  await Keychain.resetGenericPassword({ service: KEYCHAIN_SERVICE });
}

// Preserve Phase 186 signature; stays useful if a future lib wants a pre-computed headers object.
export async function applyAuth(headers: Record<string, string>): Promise<Record<string, string>> {
  const key = await getApiKey();
  return key ? { ...headers, 'X-API-Key': key } : headers;
}
```

### Context / hook wiring

`ApiKeyProvider` loads the key from Keychain on mount (async) and exposes `{apiKey, setApiKey, clearApiKey, isLoading}` via Context. The `useApiKey()` hook is a thin wrapper around `useContext(ApiKeyContext)`. No Zustand, no Redux — per ADR-003, state mgmt deferred.

On `setApiKey(newKey)`, the provider writes through to Keychain + updates local state. Components re-render with the new key and any cached API responses stay; caller code should re-fetch after key change (HomeScreen does this explicitly via a refresh button).

### Error surfacing

`openapi-fetch` returns `{data, error}` tuple. On non-2xx, `error` is the parsed response body, which per Phase 183 is always an RFC 7807 `ProblemDetail`. `src/api/errors.ts` provides:

```ts
export interface ProblemDetail {
  type: string;
  title: string;
  status: number;
  detail?: string | null;
  instance?: string | null;
  request_id?: string | null;
}

export function isProblemDetail(x: unknown): x is ProblemDetail { /* narrow */ }

export function formatProblemDetail(p: ProblemDetail): string {
  return p.detail ? `${p.title}: ${p.detail}` : p.title;
}
```

Screen-level error handling is small inline conditionals; no global error boundary in Phase 187 (if one is needed later, it's a separate phase).

### patch-package formalization

Install:
```bash
npm install --save-dev patch-package postinstall-postinstall
```

Regenerate the patch from the current in-place edit:
```bash
npx patch-package react-native-ble-plx
```

This produces `patches/react-native-ble-plx+3.5.1.patch`. Commit it. Add to `package.json`:
```json
"scripts": {
  "postinstall": "patch-package"
}
```

Now every `npm install` auto-applies the patch. Phase 186's fragile in-place edit is durable.

## Key Concepts

- **`openapi-fetch`** — ~1kb runtime typed-fetch wrapper. Zero runtime codegen. Paths + methods are typed from `openapi-typescript` output.
- **`openapi-typescript`** — CLI + library that reads an OpenAPI 3.1 JSON/YAML spec and emits TypeScript types for paths, methods, params, request bodies, responses.
- **`react-native-keychain`** — Android Keystore / iOS Keychain Services wrapper. `getGenericPassword({service})` / `setGenericPassword(user, pass, {service})` / `resetGenericPassword({service})`. Service string partitions entries by app domain.
- **`react-native-config`** — exposes `.env` vars to both JS (via `Config.API_BASE_URL`) and native code (Android: `BuildConfig.API_BASE_URL` / iOS: `[RNCConfig envFor:@"API_BASE_URL"]`). Phase 186 already installed it.
- **React Context** — singleton-state pattern without a state-management library. `ApiKeyProvider` wraps the tree; `useApiKey()` reads current key + setters.
- **`patch-package`** — reads a git-diff-style patch file from `patches/` and applies it to `node_modules/` on postinstall. Canonical workaround for library bugs that haven't been upstreamed.
- **Android Keystore** — hardware-TEE-backed key storage on most Android devices from 6.0+. Keychain-backed-keychain entries survive app updates; cleared on app uninstall.
- **RFC 7807 Problem Details** — backend's error envelope (Phase 175). `{type, title, status, detail?, instance?, request_id?}`.

## Risks

- **Backend must be running** for `npm run refresh-api-schema`. If not, the refresh script should fail loudly, not silently. Script catches fetch errors → clear error message.
- **`react-native-config` env var gotcha** — after editing `.env`, Android requires a clean rebuild to pick up new values (it bakes them into `BuildConfig` at compile time, not at runtime). Documented in the smoke-test section.
- **Keychain mocking in Jest** — must mock the native module with a clean shape, or tests hang. Will use `jest.mock('react-native-keychain', ...)` with an in-memory store.
- **openapi-typescript output drift on Windows** — potential CRLF noise. Mitigated via `.gitattributes` entry for `*.ts` if diffs turn out noisy. Not adding preemptively.
- **New Arch + react-native-keychain** — keychain's latest versions have partial New Arch support. We're on New Arch OFF (ADR-002), so this is moot. Flagging for future ADR-002 reversal.
- **Rate limiting during smoke test** — backend has per-tier rate limits; Phase 186 smoke test fixtures set budgets to 9999/min via env vars. Mobile dev hits the real limits (30/min anonymous, 100/min individual). Should be fine for manual smoke; noted as non-blocker.
- **The `npm install --save-dev patch-package` triggers its own postinstall** on a machine that doesn't have `patches/` yet. Safe no-op, but order matters: install dev deps → create patches dir → generate patch → commit.
- **ble-plx patch regeneration** — `npx patch-package react-native-ble-plx` diffs against the registry-version file. Since we've already modified the in-place file, this should produce a correct patch. Verify by inspecting the patch content before committing.
